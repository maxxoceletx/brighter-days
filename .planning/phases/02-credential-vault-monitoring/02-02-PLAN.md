**Header:**
"Alert Architecture Specification — Credential Expiry & Re-Credentialing Monitoring"
Purpose: "This document specifies the alerting system for credential expiry, CAQH re-attestation, and payer re-credentialing deadlines. It is a build specification for Phase 4 (TouchDesigner dashboard) and Phase 5 (n8n automation). Phase 2 produces this spec only — no working automation is built."

**Section 1: Credential Expiry Alerts**
- Trigger conditions: `SELECT * FROM credential_alert_queue WHERE alert_level IN ('ALERT_90', 'ALERT_60', 'ALERT_30', 'ALERT_7', 'EXPIRED')` — polled daily at 7:00 AM Pacific
- Delivery targets: BOTH admin@valentinaparkmd.com AND Maxi's email. BOTH Google Calendar event + email (belt and suspenders per CONTEXT.md)
- Cadence: 90/60/30/7 days before expiry. Alerts stop once task marked complete.
- Deduplication: One alert per credential per alert level per day. If ALERT_90 fired yesterday, don't fire again until alert level transitions (ALERT_90 -> ALERT_60).
- Alert payload: exact format from 02-RESEARCH.md (subject line, body with credential name/number/expiry/days remaining/renewal portal/1Password reference/action)
- Completion gate: credential.expiry_date updated to future date AND credential.status = 'ACTIVE'. Polling checks this before sending.

**Section 2: CAQH Re-Attestation Alerts (Special Case)**
- WHY THIS IS SPECIAL: One missed 120-day cycle can silently suspend all 17 payer contracts simultaneously. This is the highest-risk single credential in the practice.
- Trigger: credential_alert_queue where credential_name LIKE 'CAQH%'
- Alert cadence: Same 90/60/30/7 framework, but against the 120-day cycle. The 90-day alert fires 30 days into the cycle (120-90=30 days after last attestation) — which is the correct early warning.
- Alert payload: exact URGENT format from 02-RESEARCH.md with CAQH-specific language ("All 17 Payer Contracts at Risk"), link to proview.caqh.org, estimated time (5-20 min), step-by-step re-attestation instructions.
- Completion gate: Valentina re-attests in CAQH. Maxi updates credentials table: new expiry_date = CURRENT_DATE + 120, new issue_date = CURRENT_DATE. Alert stops.

**Section 3: Payer Re-Credentialing Alerts**
- Trigger: `SELECT * FROM payer_credentialing_alerts WHERE recred_urgency IN ('DUE_SOON', 'OVERDUE')`
- Alert window: 180 days before re-credentialing deadline (earlier than credential alerts because re-credentialing requires 90-day advance submission)
- Alert payload: Payer name, days until re-cred, credentialing rep contact (if known), portal URL, estimated vs confirmed flag, note about contacting Mary (credentialing agent) if re-cred process unclear
- For estimated dates: alert includes "(estimated — verify actual date with payer)" disclaimer

**Section 4: Implementation Stack**
- Automation engine: n8n (self-hosted, already in Phase 4/5 tech stack)
- Data source: Supabase views (credential_alert_queue, payer_credentialing_alerts) via Supabase REST API
- Polling interval: Daily at 7:00 AM Pacific
- Email delivery: Google Workspace SMTP or n8n email node
- Calendar integration: Google Calendar API (create/update events for expiry dates)
- TouchDesigner display: TD polls same Supabase views for real-time dashboard rendering — n8n handles push notifications, TD handles pull display

**Section 5: Data Flow Diagram**
ASCII art showing: Supabase views -> n8n polls daily -> email + calendar. Supabase views -> TD polls on refresh -> dashboard display. Human completes renewal -> updates Supabase -> alert stops.

**Section 6: Edge Cases & Failure Modes**
- What if n8n is down: Alerts don't fire. TD dashboard still shows credential status from views. Calendar events already created still remind.
- What if CAQH expiry_date is NULL: No alert fires. The credential-seed.sql has a note requiring Valentina to verify and update. Include an explicit "NULL expiry = silent failure" warning in the spec.
- What if payer recred_is_estimated = true: Alert fires but includes disclaimer. Better to alert on estimated date than miss actual deadline.
- What if credential is renewed but Supabase not updated: Alert continues firing. Include "Update Supabase after every renewal" in the spec's operational instructions.

Format: Clean markdown sections with SQL code blocks for trigger queries, ASCII for data flow, tables for payload formats. Professional tone — this is a handoff document for builders.
