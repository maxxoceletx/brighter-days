**Section 9: Panel 7 — Action Buttons (DASH-07)**
- Purpose: functional buttons that trigger real actions — send emails, run checks, generate reports. This is what makes the dashboard a command center, not just a display.
- Container COMP: panel_actions (bottom row, left half)
- Four action buttons (all four from CONTEXT.md locked decision):

  **Button 1: CAQH Re-Attestation Check**
  - Label: `[ CAQH CHECK ]`
  - Confirmation tier: IMMEDIATE (read-only check, no state modification)
  - n8n webhook: POST to `https://{n8n_host}/webhook/caqh-recheck`
  - Payload: `{ "action": "caqh_recheck", "triggered_by": "dashboard_button", "triggered_at": "{ISO timestamp}" }`
  - n8n workflow: logs into CAQH ProView (or scrapes attestation status), checks days until next attestation deadline, writes result to automation_log + updates credentials table CAQH row if new data found
  - Result displayed: inline badge transitions IDLE → RUNNING → DONE/FAILED. If DONE, tooltip shows "Last attested: {date}, Next due: {date}". If FAILED, shows error message.

  **Button 2: Generate Compliance Report**
  - Label: `[ COMPLIANCE REPORT ]`
  - Confirmation tier: IMMEDIATE (read-only, generates document)
  - n8n webhook: POST to `https://{n8n_host}/webhook/compliance-report`
  - Payload: `{ "action": "compliance_report", "triggered_by": "dashboard_button", "triggered_at": "{ISO timestamp}" }`
  - n8n workflow: queries credential_alert_queue + payer_credentialing_alerts + obligations, generates a formatted markdown/PDF report, saves to a designated folder (Google Drive or local), writes file URL to automation_log.details
  - Result: inline badge shows DONE + link to generated report in automation tracker

  **Button 3: Trigger Payer Status Checks**
  - Label: `[ PAYER STATUS ]`
  - Confirmation tier: IMMEDIATE (read-only checks across portals)
  - n8n webhook: POST to `https://{n8n_host}/webhook/payer-status-check`
  - Payload: `{ "action": "payer_status_check", "triggered_by": "dashboard_button", "triggered_at": "{ISO timestamp}" }`
  - n8n workflow: for each payer with a portal URL in payer_tracker, checks credentialing status (where automatable). Writes summary to automation_log. Updates payer_tracker rows if new contract info found.
  - Note: many payer portals require manual login — spec should note which portals are automatable vs which require human action. The initial implementation may only check a subset.

  **Button 4: Send Patient Communications**
  - Label: `[ SEND COMMS ]`
  - Confirmation tier: CONFIRM REQUIRED (sends external email — state modification)
  - On button press: opens a confirmation dialog overlay (Container COMP). Dialog shows:
    - Communication type dropdown: Intake Packet / Appointment Reminder / Consent Forms
    - Recipient: text field for patient first name + email (manually entered — no patient list on dashboard for PHI reasons)
    - Preview of email content (template-based)
    - `[ CONFIRM SEND ]` and `[ CANCEL ]` buttons
  - On confirm: POST to `https://{n8n_host}/webhook/send-communication`
  - Payload: `{ "action": "send_communication", "comm_type": "{type}", "recipient_email": "{email}", "recipient_first_name": "{name}", "triggered_by": "dashboard_button", "triggered_at": "{ISO timestamp}" }`
  - n8n workflow: uses SendGrid or email node to send templated email, writes result to automation_log
  - PHI note: email address entered in dashboard is transient (not stored in Supabase). Only the action record (type + timestamp + status) is logged.

- **Inline status badges** (per button, per CONTEXT.md):
  - State machine: IDLE → RUNNING → (DONE | FAILED)
  - Visual states (ASCII art):
    - IDLE: `[ CAQH CHECK ]` in GREEN text
    - RUNNING: `[ CAQH CHECK ... ]` with spinning indicator (Pattern CHOP driving character cycle `|/-\`)
    - DONE: `[ CAQH CHECK OK ]` in GREEN with brief flash animation
    - FAILED: `[ CAQH CHECK ERR ]` in RED, holds for 30 seconds then returns to IDLE
  - State driven by: latest row in automation_log WHERE action_type = '{button_action}' — Python callback on automation_log table data reads status column

- **Confirmation dialog spec** (for CONFIRM REQUIRED actions):
  - Container COMP: confirm_dialog (overlay, centered on screen, semi-transparent dark background behind)
  - Contains: Text TOP (action description), input fields (Button COMPs styled as form elements), Confirm button (GREEN), Cancel button (RED)
  - Cancel returns to dashboard without action
  - Confirm triggers webhook and closes dialog

**Section 10: Panel 8 — Automation Process Tracker (DASH-08)**
- Purpose: full history of every automation triggered from the dashboard — what ran, when, what happened
- Container COMP: panel_automation (bottom row, right half)
- Data source: Supabase automation_log table, ordered by triggered_at DESC
- Display format: scrolling ASCII log (newest at top), each entry showing:
  ```
  [HH:MM] ACTION_TYPE .............. STATUS
  [14:32] CAQH CHECK ............... OK
  [14:30] COMPLIANCE REPORT ........ OK (report-2026-03-01.pdf)
  [09:15] PAYER STATUS ............. FAILED (timeout)
  [09:00] SEND COMMS ............... OK (intake to Alex)
  ```
- Status column color: OK in GREEN, FAILED in RED, RUNNING in AMBER with spinning indicator
- Expandable detail: clicking a log entry opens an overlay showing:
  - Full timestamp (ISO format)
  - Action type and trigger source (dashboard_button / scheduled / auto)
  - Duration (completed_at - triggered_at)
  - Details field (error message for failures, file URL for reports, recipient info for comms)
- Scrolling behavior: panel shows last 20 entries by default. Scroll up to see older. Auto-scrolls to top when new entry appears.
- Auto-refresh: polls automation_log on the global refresh cycle (15-30 min) BUT also does a quick poll 5 seconds after any button press (to show the RUNNING → DONE/FAILED transition promptly)
- ASCII aesthetic: log entries use dot-leader fill between action name and status (like the example above). Monospace alignment ensures clean columns.
